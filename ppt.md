# PPT 制作思路

> 适用于项目答辩，建议控制在 **15～20 张**幻灯片，答辩时长 10～15 分钟。

---

## 整体叙事逻辑

答辩 PPT 的核心叙事线要像讲故事一样，遵循：

**痛点 → 解决方案 → 技术实现 → 亮点 → 效果验证 → 总结展望**

切忌把 PPT 做成代码说明书，要让评委一眼看懂"你解决了什么问题、怎么解决的、效果好不好"。

### 第 3 张：背景与问题（为什么要做）

**内容要素**
- 现实痛点：大量文档（合同、报告、试卷、论文）以 PDF/图片形式存在，人工处理效率极低
- 现有方案的不足：
  - 普通 OCR 只能提取文字，丢失结构信息（标题/段落/表格/公式）
  - 商业工具（如 Adobe）价格高昂，不可定制
  - 现有开源工具对中文文档支持弱、对混合排版（图文并茂）支持差
- 应用需求：RAG 知识库构建、文档数字化归档、合同自动解析等场景对结构化文档解析需求迫切

**视觉建议**
- 左：一张含表格+公式+图片的复杂中文文档截图
- 右：显示普通 OCR 输出的混乱结果（纯文字，结构丢失）

### 第 4 张：系统目标与定位

**内容要素**
- 输入：PDF 文件 / 扫描图片（JPG/PNG/BMP）
- 输出：结构化 HTML + 标准 JSON（含元素类型、像素坐标、置信度）
- 支持识别 8 类文档元素：
  - 多级标题（h1/h2/h3）
  - 正文段落（p）
  - 表格（table）
  - 图片（image）
  - 数学公式（formula）
  - 页眉 / 页脚 / 页码

**视觉建议**
- 用图标+文字列出 8 类元素，每类一个小图标

### 第 5 张：系统总体架构（核心）

**内容要素**
使用流程图展示 5 个处理阶段：

```
输入(PDF/图片)
    ↓
Stage 1: 预处理（图像增强）
    ↓
Stage 2: 元素检测（PPStructure + OCR 三级保障）
    ↓
Stage 3: 内容识别（文本/表格/公式/图片 分路处理）
    ↓
Stage 4: 逻辑重建（多栏/层级/段落/页码）
    ↓
Stage 5: 结构化输出（HTML + JSON）
```

**视觉建议**
- 横向或纵向流程图，每个 Stage 用不同颜色的方块
- 重点标注第 2、3、4 阶段（技术含量最高）

### 第 6 张：Stage 1 — 图像预处理

**内容要素**
- PDF 转图像：使用 pdf2image（300 DPI）逐页转换
- 图像增强流程：
  1. 高斯去噪（3×3 核）
  2. CLAHE 对比度增强（clipLimit=2.0）
  3. 尺寸归一化（最大宽度 2000px）
- 保留彩色信息（不二值化，为后续表格/图片识别保留颜色）

**视觉建议**
- 并排展示：原图（左）→ 增强后（右），直观对比效果

---

### 第 7 张：Stage 2 — 三级保障元素检测（重点）

**内容要素**
核心创新点，重点讲解：

| 级别 | 触发条件 | 处理策略 |
|---|---|---|
| 第一级 | PPStructure 成功且文本 ≥ 3 | 直接使用版面分析结果 |
| 第二级 | PPStructure 成功但文本 < 3 | OCR 补充扫描 + IoU 去重合并 |
| 第三级 | PPStructure 失败/无结果 | 完全降级为纯 OCR + 位置规则 |

- 特别说明：PPStructure 的 figure 误判修复——对内含文字的"图片区域"自动展开为逐行文本元素

**视觉建议**
- 用三个分支的决策树图来展示三级降级逻辑

---

### 第 8 张：Stage 3 — 多模态内容识别（重点）

**内容要素**
按元素类型分四条识别链路：

- **文本**：二次 OCR 补充（填充空白内容）+ 置信度过滤标记
- **表格**：PPStructure 表格结构识别 → HTML `<table>`（失败时 OCR 降级）
- **公式**：Mathpix API → LaTeX `$$...$$`（未配置时 OCR 降级为 `$...$`）
- **图片**：裁切区域保存至 `output/images/`，路径写入 content 字段

**视觉建议**
- 四个横向并排的小卡片，每个卡片展示一种元素类型的处理链路

---

### 第 9 张：Stage 4 — 逻辑关系重建（重点）

**内容要素**
四步处理流程（依次执行）：

1. **页码识别**：正则匹配（纯数字、"第N页"、"N/M"等），结合位置过滤，标记为 `page_number`
2. **标题层级推断**：按 bbox 高度（字体大小近似）动态分层，无需手工设阈值，自适应不同文档
3. **段落聚合**：行间距 > 平均行高×1.5 则分段，写入 `paragraph_id`，直接服务 RAG chunking
4. **多栏排序**：分析元素中心 x 坐标分布判断双栏，栏优先排序（左栏从上到下，再读右栏）

**视觉建议**
- 用双栏文档的示例，展示"错误排序（全局y排序）"vs"正确排序（栏优先）"的对比

---

### 第 10 张：Stage 5 — 标准化输出

**内容要素**
两种输出格式并列展示：

**HTML 输出特性**
- 完整内嵌 CSS 样式，浏览器直接打开
- 每个元素携带 `data-bbox="x1,y1,x2,y2"` 像素坐标
- 低置信度元素自动标记（`opacity: 0.6`）

**JSON 输出特性**
- 标准化结构，含 index、page_num、element_type、bbox、content、confidence
- 附加语义字段：paragraph_id、formula_engine、image_path

**视觉建议**
- 左侧：HTML 渲染效果截图
- 右侧：JSON 代码片段（选 3～4 个字段）

### 第 11 张：技术创新点汇总（重点）

**内容要素**
用卡片/表格集中展示 5 大创新点，简洁有力：

| 创新点 | 解决的问题 | 技术手段 |
|---|---|---|
| 三级保障检测架构 | PPStructure 误判/不可用导致内容丢失 | figure 文字展开 + IoU 去重合并 |
| 动态标题分层 | 不同文档字号不同，固定阈值失效 | bbox 高度自适应聚类 |
| 多栏感知排序 | 双栏文档全局 y 排序阅读顺序错乱 | x 坐标双峰检测 + 栏优先排序 |
| 段落语义聚合 | 文本行是最小粒度，无段落概念 | 行间距阈值 × 1.5，输出 paragraph_id |
| 公式双链路识别 | 复杂公式 OCR 识别效果差 | Mathpix API 优先 + 本地 OCR 降级 |

### 第 14 张：技术选型说明

**内容要素**
为什么选择这些技术：

| 技术 | 版本 | 选择理由 |
|---|---|---|
| PaddleOCR | 2.7.0 | 国产最优中文 OCR，中文识别率业界领先 |
| PPStructure | 同上 | PaddleOCR 内置版面分析，表格/图片/公式一体化 |
| OpenCV | 4.6 | 高性能图像处理，CLAHE 增强效果好 |
| Python 3.9 | — | 生态成熟，paddle 官方推荐 |
| Mathpix API | — | 全球最准的公式 LaTeX 识别服务（可选） |

### 第 15 张：系统局限性与不足（体现思考深度）

**内容要素**
主动列出不足，展示技术诚信：

- PPStructure 版面模型对非标准版面（手写文档、古籍、高度定制化排版）识别效果有限
- 当前多栏检测只支持单栏/双栏，三栏及以上布局暂未支持
- 公式识别在无 Mathpix 时质量有限，复杂数学公式可能识别为普通文本
- 处理速度受模型推理影响，大批量文档需引入并发/批处理优化
- 暂不支持手写体文档的专项识别（需额外训练模型）

### 第 16 张：应用场景与扩展方向

**内容要素**
- 当前应用场景：
  - RAG 知识库文档预处理（JSON 输出 → 向量化 → 检索）
  - 企业文档数字化归档
  - 合同/报告自动解析系统的后端引擎
- 未来扩展方向：
  - 接入大模型对识别内容做语义校正和补全
  - 支持批量处理与 Web API 化部署（FastAPI）
  - 扩展手写体文档支持
  - 输出格式扩展（Markdown、Word、结构化数据库）

## 设计风格建议

- **配色**：以蓝白为主色调，强调色用天蓝（#1E90FF），辅以浅灰背景
- **字体**：标题用思源黑体加粗，正文用思源宋体，代码用等宽字体（Consolas/Source Code Pro）
- **动画**：尽量少用，流程图可用逐步显示，其余保持静态
- **图表**：架构图、流程图一律自己绘制，不要截图拼凑，保持统一风格
- **每页字数**：正文内容不超过 80 字，多用图表和数据说话
