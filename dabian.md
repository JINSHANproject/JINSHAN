# 项目答辩详细材料

> 多模态文档识别系统（基于 PaddleOCR / PPStructure）
> 本文档供答辩备用，包含项目深度解析和高频问答，请充分熟悉后上场。

---

## 一、项目背景与立项动机

### 1.1 现实问题

在企业和学术场景中，大量有价值的文档以 **PDF 扫描件或图片**的形式存在，例如：

- 合同文件（含表格、签名、盖章）
- 学术论文（含公式、图表、参考文献）
- 教材试卷（含数学公式、选项表格）
- 企业报告（含多栏排版、图片、数据表格）

这类文档的问题在于：

1. **无法直接检索**：PDF 扫描件本质是图片，无法 Ctrl+F 搜索
2. **结构信息丢失**：传统 OCR 只能提取一串文字，标题/段落/表格/公式的区别全部丢失
3. **人工成本高**：人工整理一份 20 页复杂文档往往需要数小时
4. **下游处理困难**：RAG（检索增强生成）、文档问答等 AI 应用需要结构化输入，而非一大段混排文字

### 1.2 现有方案的不足

| 方案 | 优点 | 缺点 |
|---|---|---|
| 普通 OCR（Tesseract 等） | 开源免费 | 只出文字，无结构，中文效果差 |
| Adobe Acrobat Pro | 功能完整 | 价格昂贵，不可编程集成 |
| AWS Textract / Azure Form | 云端高精度 | 付费，数据上云有安全顾虑 |
| 早期 PaddleOCR | 中文好 | 仅文字，无版面分析 |

### 1.3 项目定位

本项目旨在构建一套**完全开源、本地运行、可编程集成**的文档多模态解析系统，输出**结构化 HTML + 标准 JSON**，支持 8 类文档元素的精确识别与定位，可直接对接 RAG 流水线。

## 二、系统整体设计

### 2.1 设计原则

1. **完整性**：不遗漏文档中的任何有效信息（文字、表格、公式、图片均要处理）
2. **鲁棒性**：在版面分析模型失效时能优雅降级，保证系统不崩溃
3. **可用性**：一条命令即可运行，输出可直接在浏览器查看
4. **可扩展性**：模块化设计，每个阶段可独立替换或升级

### 2.2 处理流水线

系统由 **5 个串行阶段**组成，每阶段职责单一：

```
Stage 1: 预处理         core/preprocess.py
Stage 2: 元素检测       core/element_detect.py
Stage 3: 内容识别       core/content_recognize.py
Stage 4: 逻辑重建       core/logic_rebuild.py
Stage 5: 结构化输出     core/output_generator.py
```

各阶段之间通过统一的**元素字典列表**（`List[Dict]`）传递数据，格式如下：

```python
{
    "element_type": "p",            # 元素类型
    "bbox": [100, 140, 800, 165],   # 像素坐标 [x1,y1,x2,y2]
    "content": "这是一段正文。",     # 识别内容
    "page_num": 1,                  # 所在页码
    "extra": {
        "confidence": 0.9923,       # 置信度
        "paragraph_id": 2,          # 段落归属 ID
        "raw_type": "text",         # PPStructure 原始类型
        "ocr_lines": [...],         # OCR 行详情
    }
}
```

### 2.3 支持的元素类型（8 类）

| 内部类型 | 来源 | HTML 标签 | JSON 附加字段 |
|---|---|---|---|
| `h1` / `h2` / `h3` | title（按字体高度自动分层） | `<h1>~<h3>` | — |
| `p` | text | `<p>` | `paragraph_id` |
| `table` | table | `<div class="doc-table-wrap">` | — |
| `image` | figure | `<figure>` | `image_path` |
| `formula` | formula/equation | `<div class="formula">` | `formula_engine` |
| `header` | header | `<div class="doc-header">` | — |
| `footer` | footer | `<div class="doc-footer">` | — |
| `page_number` | 正则推断 | `<span class="page-num">` | — |

## 三、各模块技术细节

### 3.1 Stage 1：图像预处理（`core/preprocess.py`）

**PDF 处理**
- 使用 `pdf2image`（基于 poppler）将 PDF 每页渲染为 300 DPI 的 PNG
- 对 RGBA/LA 模式（透明背景）自动合成白色底图，防止透明区域干扰 OCR

**图像增强流程**
```
原始图像
  → 高斯模糊（3×3 核，sigma=0）     # 去除高频噪点
  → 转灰度
  → CLAHE 自适应直方图均衡化          # 增强局部对比度，参数 clipLimit=2.0, tileGridSize=(8,8)
  → 转回 BGR 三通道                  # 保留颜色信息，不二值化
  → 等比缩放（宽度超 2000px 时）      # 防止超大图像拖慢推理
  → 保存为 tmp/enhanced_*.png
```

**为什么不二值化？**
二值化会损失表格边框颜色、图片颜色信息，影响 PPStructure 对表格和图片区域的识别。保留三通道彩色图，让下游模型自行处理。

### 3.2 Stage 2：元素检测（`core/element_detect.py` + `utils/ocr_helper.py`）

#### 主引擎：PPStructure

PPStructure 是 PaddleOCR 内置的版面分析系统，调用方式：

```python
structure = PPStructure(table=True, ocr=True, lang='ch', use_gpu=False)
result = structure(img)
# result 是一个列表，每项形如：
# {
#   'type': 'text' | 'title' | 'table' | 'figure' | 'formula' | 'header' | 'footer',
#   'bbox': [x1, y1, x2, y2],
#   'res': [{'text': '...', 'confidence': 0.99, 'text_region': [[x,y],[x,y],...]}],
#   'img': <numpy array>   # 区域裁切图像
# }
```

#### figure 误判修复（关键）

PPStructure 模型在以下场景会将文字区域误标为 `figure`：
- 含图标、徽标的信息图（Infographic）
- 演示文稿（PPT）截图
- 含彩色背景的文字块

**修复逻辑（`parse_structure_result()` 中实现）**：
```
若 region['type'] == 'figure'
  且 region['res'] 中 OCR 文本行 >= 2 条
  → 将该区域展开为逐行独立 p 元素（每行附行级 bbox）
  → 不作为图片处理
```

这确保了即使 PPStructure 误判，所有文字内容也不会丢失。

#### 三级保障降级策略（`detect_elements()` 中实现）

```
PPStructure 运行成功？
  ├─ 否 → [Level 3] 完全使用纯 OCR + 位置规则
  └─ 是 → 提取的文本元素数 >= 3？
         ├─ 是 → [Level 1] 直接使用 PPStructure 结果
         └─ 否 → [Level 2] 追加纯 OCR 扫描
                           → IoU 去重（阈值 0.5）
                           → 合并到 PPStructure 结果
```

**IoU 去重**：对每条 OCR 补充行计算与已有元素的交并比（IoU），超过阈值则视为重复，不添加，防止同一区域出现两次识别结果。

### 3.3 Stage 3：内容识别（`core/content_recognize.py`）

按 element_type 分四条链路并行处理：

#### 文本识别（`recognize_text_elements`）
- 检查 `content` 是否已有内容，有则跳过
- 对空 content 的文本元素，裁切原始图像区域进行二次 OCR 填充
- 低置信度（< 0.5）的元素标记 `extra.low_confidence = True`

#### 表格识别（`recognize_table_elements`）
```
裁切表格区域（加 4px padding 防边缘截断）
  → 调用 PPStructure 对裁切图识别
    → 成功：取 type=table 的 res，写入 HTML <table>
    → 失败：对区域做纯 OCR，逐行拼成简单表格
```

#### 公式识别（`recognize_formula_elements`）
```
配置了 MATHPIX_APP_ID？
  ├─ 是：将公式区域图像 base64 编码 → 调用 Mathpix API → 返回 LaTeX
  │      结果格式：$$E=mc^2$$
  └─ 否：对区域做 OCR → 包裹 $...$ 标记（降级）
```

#### 图片处理（`recognize_image_elements`）
- 裁切图片区域，保存至 `output/images/page{N}_img{M}.png`
- `content` 字段记录相对路径 `images/page1_img1.png`

### 3.4 Stage 4：逻辑重建（`core/logic_rebuild.py` + `utils/coordinate.py`）

按固定顺序执行四步：

#### 步骤 1：页码识别（`identify_page_numbers`）
正则匹配如下格式，同时判断位置（页面顶部 8% 或底部 8%）或尺寸（高度 < 30px）：
- 纯数字：`\d+`
- 分式：`\d+/\d+`（如"3/10"）
- 中文：`第\s*\d+\s*页`
- 装饰式：`- 5 -`

#### 步骤 2：标题层级推断（`infer_heading_levels`）
```python
title_heights = [bbox[3]-bbox[1] for e in title_elements]
# 最大高度 → h1，次大 → h2，其余 → h3
# 完全数据驱动，无硬编码阈值
```

**优势**：对 A4 论文、A3 海报、PPT 截图等不同尺寸文档均适用，不需要调参。

#### 步骤 3：段落聚合（`assign_paragraph_hierarchy`）
```python
avg_line_height = mean([bbox[3]-bbox[1] for p in p_elements])
# 两行间距 > avg_line_height * 1.5 → 新段落
# 写入 extra['paragraph_id']
```

**为什么重要**：RAG 系统需要把文档切分为语义完整的"块"（chunk）。`paragraph_id` 相同的行属于同一段落，可以一起输入向量模型。

#### 步骤 4：多栏感知排序（`sort_elements_multicolumn`）
```python
# 1. 计算所有文本元素中点 x 坐标
# 2. 与页面中线比较，若左右各超 25% → 判定为双栏
# 3. 双栏：左栏元素按 y 排序 + 右栏元素按 y 排序，left 在前
# 4. 单栏：全局 (y, x) 排序
```

### 3.5 Stage 5：结构化输出（`core/output_generator.py`）

**HTML 输出**
- 每个元素生成对应 HTML 标签，带 `data-bbox` 属性
- 低置信度元素带 `data-low-confidence="true"`，CSS 设置 `opacity: 0.6`
- 内嵌完整 CSS，支持表格样式、公式字体、响应式布局
- 公式内容不做 HTML 转义（保留 `$$` 标记，前端可接 MathJax 渲染）

**JSON 输出**
```json
{
  "elements": [
    {
      "index": 0,
      "page_num": 1,
      "element_type": "h1",
      "bbox": [100, 50, 700, 90],
      "content": "文档标题",
      "confidence": 0.9977,
      "paragraph_id": 1
    }
  ],
  "total": 12
}
```

## 四、技术创新点详解

### 创新点一：三级保障检测架构 + figure 误判自动修复

**解决的问题**：PPStructure 版面分析模型对非标准版面（信息图、PPT截图、彩色背景文字块）容易将文字区域误标为 `figure`，导致所有文字内容被丢弃，输出空文档。

**技术方案**：
1. 在 `parse_structure_result()` 中对每个 `figure` 区域检查 `res` 字段，若含 ≥2 条文本行则自动展开为逐行 `p` 元素
2. 在 `detect_elements()` 中增加第二级保障：PPStructure 有结果但文本量不足时，补充 OCR 扫描并通过 IoU 去重合并

**实际效果**：在测试图片（PPT截图风格的信息图）上，从原来识别 0 条有效文字→修复后准确识别 7 条，置信度均 > 0.97。

### 创新点二：自适应标题分层

**解决的问题**：传统方案用固定字号（如 > 20px 为标题）区分标题和正文，但不同文档字号差异巨大，固定阈值失效。

**技术方案**：收集当前文档所有 `title` 元素的 bbox 高度，用数据驱动的聚类分层（高度最大 → h1，次大 → h2，其余 → h3），无任何硬编码参数。

### 创新点三：多栏布局感知排序

**解决的问题**：双栏文档如果按全局 y 坐标排序，会出现"左栏第一行→右栏第一行→左栏第二行→..."的错误交叉顺序，无法正常阅读。

**技术方案**：
1. 统计元素中心 x 坐标的分布，用页面中线±10% 作为过渡区
2. 若左右各有 ≥25% 的元素，判定为双栏
3. 双栏时先对左栏按 y 排序，再对右栏按 y 排序，保证正确阅读顺序

### 创新点四：段落语义聚合

**解决的问题**：OCR 输出以"行"为最小单位，同一段落的多行文字是独立元素，RAG 系统无法识别段落边界。

**技术方案**：自适应行间距阈值（平均行高 × 1.5），将连续行聚合为段落组，写入 `paragraph_id`，RAG 系统可直接按此字段分块。

### 创新点五：公式识别双链路

**解决的问题**：PaddleOCR 对数学公式识别效果不佳，复杂公式（积分、矩阵、分式）识别结果混乱。

**技术方案**：
- 高精度链路：接入 Mathpix API，将公式图像 base64 上传，返回标准 LaTeX
- 降级链路：无 API 时用 OCR 识别并包裹 `$...$` 标记，至少保留公式区域的文字信息
- 两条链路自动切换，通过环境变量配置，无需修改代码

## 五、评委高频问答（详细版）

### 模块一：背景与选题类

**Q1：为什么选择 PaddleOCR 而不是 Tesseract 或其他 OCR 引擎？**

A：选择 PaddleOCR 的主要原因有三点：
1. **中文识别效果**：PaddleOCR 对中文的识别准确率在开源方案中排名前列，支持繁简混排、印刷体与手写体混合等复杂场景
2. **PPStructure 集成**：PaddleOCR 内置 PPStructure 版面分析模块，可在一个框架内同时完成文本检测和版面结构识别，无需集成多个独立工具
3. **工程成熟度**：由百度飞桨团队维护，文档完整，社区活跃，生产环境有大量验证案例

Tesseract 虽然历史悠久，但对中文支持较弱，且没有版面分析能力；商业 OCR API（如腾讯文字识别）精度高但有费用和数据安全问题。

---

**Q2：你们的系统和市面上的 PDF 解析工具（如 PyMuPDF、pdfplumber）有什么区别？**

A：核心差异在于处理对象不同：
- PyMuPDF、pdfplumber 等工具处理的是**数字原生 PDF**（有文字图层的 PDF），直接提取文字坐标，速度快但无法处理扫描件
- 本系统针对的是**扫描件/图片文档**，先通过 OCR 和版面分析从图像中理解内容，因此适用范围更广
- 此外，本系统输出的是**结构化元素列表**（含类型、坐标、置信度、段落ID），而非一维文字字符串，更适合下游 AI 应用

两类工具不是替代关系，可以组合使用：数字 PDF 用 pdfplumber 直接提取，扫描件用本系统处理。

---

**Q3：你们解决的核心技术挑战是什么？**

A：最核心的挑战有两个：

第一，**版面理解**。文档不只是文字，还有表格、图片、公式，而且不同文档的排版差异极大（单栏/双栏、横排/竖排、有无图标装饰）。PPStructure 的版面模型虽然强大，但对非标准版面（如含装饰图标的 PPT 截图）会出现误判，把文字区域标记为图片。我们通过 `res` 字段二次检测和三级降级策略解决了这个问题。

第二，**阅读顺序恢复**。文档元素在图像上的坐标不等于阅读顺序，尤其是双栏文档。我们实现了基于 x 坐标分布的多栏检测，确保双栏文档按正确的栏优先顺序输出。

---

### 模块二：技术实现类

**Q4：PPStructure 的 figure 误判问题是怎么发现的，怎么解决的？**

A：在实际测试中，我们用一张 PPT 风格的信息图作为测试文件。运行后发现 JSON 输出中 total=1 且 element_type 为 image，而图片中明显有 7 行文字，完全没有被识别出来。

排查过程：打印 PPStructure 的原始返回值，发现它将整个主体区域标记为 `figure`，但 `res` 字段里实际包含了 9 行 OCR 识别结果。问题根源在于 PPStructure 的版面分析模型——该模型在 PublayNet 数据集上训练，对中文信息图这类非标准文档的版面判断会出现偏差，把图文混排区域整体归类为图片。

解决方案：在 `parse_structure_result()` 中增加二次判断：如果 `figure` 区域的 `res` 字段包含 ≥2 条文本行，说明这是被误判的文字区域，将每条文本行展开为独立的 `p` 元素，并使用行级 bbox 而非区域大框。这个修复方案的核心思路是"相信 OCR 的子结果，而不是版面分类器的标签"。

---

**Q5：三级降级策略的设计思路是什么？为什么要设置文本元素数量阈值 3？**

A：三级策略本质上是**质量门控**：不仅检查 PPStructure 是否成功运行，还检查其输出质量是否达标。

阈值 3 的来源：从实际测试发现，PPStructure 在模型未完全加载或版面误判时，往往只返回 0～2 个区域，且这些区域通常是大面积的 `figure` 或低置信度文本块。正常情况下，一页有意义的文档至少包含 3 个以上独立的文字区域（标题+至少两段正文）。因此 3 作为"最低质量线"是合理的经验值。

第二级（OCR 补充+IoU 合并）的设计目的：避免完全抛弃 PPStructure 的成果（它可能正确识别了表格/公式等非文字元素），只补充缺失的文字部分。IoU 去重阈值 0.5 的含义是：只要新增元素与已有元素重叠超过 50%，就视为重复，不添加。

---

**Q6：CLAHE 增强的参数是怎么选的？**

A：CLAHE（对比度受限自适应直方图均衡化）的两个核心参数：

- `clipLimit=2.0`：限制对比度放大的上限。值越大，增强效果越强，但噪点也会被放大。2.0 是 OpenCV 官方推荐的平衡值，在提升对比度的同时避免过增强导致文字边缘模糊。
- `tileGridSize=(8,8)`：将图像分成 8×8 的小块分别做直方图均衡。格子越细，局部适应能力越强，但计算量越大。8×8 对 1000px 级别的图像而言，每格约 125px，足以处理局部光照不均匀（如扫描件边缘偏暗）的问题。

参数选择依据是在测试集上的主观视觉评估，对明显不均匀照明的扫描文档效果最好。

---

**Q7：多栏检测的 25% 阈值和 10% 中线过渡区是怎么来的？**

A：这两个参数通过经验值确定：

- **10% 中线过渡区**：处理跨栏元素（如横跨双栏的大标题、图片），这类元素的中心 x 坐标落在页面中间，不属于任何一栏。设置 10% 的过渡区将其排除在栏判断之外，防止误判。
- **25% 最小占比**：确保双栏的两侧都有实质内容。如果只有左侧有元素，不应判断为双栏（可能只是文档有大面积留白）。25% 是合理的最低阈值，两侧各 25% 加上中间过渡区约等于 60%，说明页面两侧都有较多元素。

如果测试的文档有特殊排版，可以在 `utils/coordinate.py` 的 `detect_columns()` 中调整这两个参数。

---

**Q8：系统的输出 JSON 中为什么同时有 `confidence` 和 `low_confidence` 字段？**

A：两个字段用途不同：

- `confidence`：精确的浮点数值（0.0～1.0），供程序精细化处理，例如可以设置不同的过滤阈值
- `low_confidence: true`：布尔标记，只在置信度低于配置阈值时出现，用于快速过滤——下游系统只需判断 key 是否存在即可，不需要写数值比较逻辑

这是一种**冗余但便利**的设计：`confidence` 完整，`low_confidence` 便捷，两者协同使用。

---

**Q9：`paragraph_id` 对 RAG 应用有什么实际价值？**

A：RAG 系统在向量化时需要将文档切成合适的"块"（chunk）。切块策略直接影响检索质量：

- 太短（按行切）：语义不完整，检索时难以匹配
- 太长（按页切）：信息密度低，向量表示模糊，也超出部分模型的输入上限

`paragraph_id` 提供了天然的语义边界：相同 `paragraph_id` 的文本行属于同一段落，一段话在语义上是完整的。下游 RAG 系统可以：

```python
# 按 paragraph_id 聚合，每个 chunk 是一个完整段落
from collections import defaultdict
chunks = defaultdict(list)
for ele in elements:
    if ele['element_type'] == 'p':
        pid = ele.get('paragraph_id', 0)
        chunks[pid].append(ele['content'])
final_chunks = [' '.join(lines) for lines in chunks.values()]
```

### 模块三：系统局限与改进类

**Q10：系统目前有哪些明显的不足？**

A：诚实回答有以下几点：

1. **版面模型局限**：PPStructure 使用的版面分析模型在 PublayNet（英文学术论文）和 CDLA（中文学术文档）数据集上训练，对这两类场景效果好，但对非标准版面（手写文档、古籍、高度定制化的企业模板）需要依靠降级策略兜底。

2. **多栏支持有限**：当前多栏检测只支持单栏和双栏，三栏及以上（如报纸排版）会退化为简单的全局排序，阅读顺序可能不正确。

3. **公式识别依赖外部 API**：本地 OCR 对复杂数学公式的识别效果有限，高质量识别依赖 Mathpix 等付费 API。

4. **速度**：单页文档约 2 秒（CPU 模式），大批量处理时没有并发优化，处理 100 页文档需要约 200 秒。

5. **手写体专项支持弱**：PaddleOCR 虽然理论上支持手写体，但在混合排版（印刷体+手写批注）场景下识别效果不稳定。

---

**Q11：如果要优化速度，你们会怎么做？**

A：速度优化有几个方向：

1. **GPU 加速**：当前系统已支持 GPU 推理（`Config.use_gpu()` 自动检测），有 GPU 环境时速度可提升 5～10 倍
2. **批处理**：多页文档同时推理，而不是逐页串行处理
3. **异步 IO**：预处理（图像增强）和模型推理可以并行，提前加载下一页图像
4. **模型量化**：使用 INT8 量化版本的 PaddleOCR 模型，推理速度提升约 2～3 倍，精度略有损失
5. **Web API 化**：用 FastAPI 将系统包装为 HTTP 服务，模型常驻内存，避免每次调用的模型加载时间（首次加载约 3～5 秒）

---

**Q12：如何评估系统的识别质量（有没有量化的评估指标）？**

A：目前使用的是定性测试（人工对比输入输出），这是系统的一个不足。

理论上可以使用以下量化指标：
- **文本识别精度**：字符级 CER（字符错误率）或词级 WER，需要人工标注的 ground truth
- **版面检测精度**：元素级别的 IoU（检测框与标注框的交并比），使用 mAP（mean Average Precision）衡量
- **端到端结构化准确率**：与标准输出 JSON 逐字段对比

实际上 PaddleOCR 官方在公开数据集（ICDAR、CTW1500 等）上有详细的 benchmark 数据，本系统的识别精度上界由底层模型决定。

---

**Q13：系统如何处理旋转的文字或倒置的扫描件？**

A：通过两个机制处理：

1. **PaddleOCR 文字方向分类**：系统初始化时开启了 `use_angle_cls=True`，该模块会对每个文字行检测方向（0°、90°、180°、270°），并在识别前自动旋转校正
2. **PPStructure 图像方向检测**：PPStructure 内部有 `image_orientation_predictor`，可以检测整页图像的旋转角度并在版面分析前校正

对于严重歪斜的扫描件（非 90° 整数倍的倾斜），当前系统没有专项处理，这是一个可以用 Hough 变换或深度学习倾斜校正模块来改进的方向。

---

**Q14：为什么选择 HTML 作为主要输出格式，而不是 Markdown 或 Word？**

A：HTML 的选择基于以下考虑：

1. **原生支持多媒体元素**：HTML 天然支持 `<table>`、`<img>`、数学公式渲染（MathJax），而 Markdown 对复杂表格和公式的支持有限，Word 则需要额外的 python-docx 库
2. **data-bbox 属性可扩展**：HTML 的 `data-*` 自定义属性可以把每个元素的像素坐标嵌入，方便前端实现原文高亮、对位标注等功能，这在 Markdown 或 Word 中很难做到
3. **浏览器直接预览**：无需任何额外工具，直接双击 result.html 即可查看结构化结果
4. **标准规范**：HTML 是国际标准，任何下游系统都可以解析

当然，如果需要 Markdown 或 Word 输出，可以基于现有的元素列表扩展 `output_generator.py` 来支持，架构上是完全可以扩展的。

---

**Q15：你们的系统与大模型（如 GPT-4V、Claude）的多模态文档理解有什么区别和联系？**

A：这是一个很好的问题，两者定位不同，互补性很强：

**区别**：
- 大模型（GPT-4V 等）是端到端的"理解"——输入文档图片，直接输出自然语言答案，擅长语义推理，但无法精确输出结构化坐标，且成本高、有隐私顾虑
- 本系统是"解析+结构化"——输出的是精确的 JSON 元素列表，含像素坐标和置信度，适合工程级别的文档处理流水线，完全本地运行

**联系**：
- 本系统的 JSON 输出可以作为大模型的**结构化输入**，给大模型提供"哪里是标题、哪里是表格"的先验信息，提升大模型在文档问答任务上的精度
- 两者可以组成"本系统解析 + 大模型理解"的完整流水线：本系统负责提取结构，大模型负责语义推理

---

**Q16：整个系统的代码量大约是多少，你们是如何组织代码的？**

A：项目共约 1650 行 Python 代码（不含注释和空行），分为三层：

- `config/`：全局配置，1 个文件，约 50 行
- `core/`：核心业务逻辑，5 个文件，约 1000 行，每个文件对应一个处理阶段
- `utils/`：工具函数，4 个文件，约 600 行

代码组织遵循**单一职责原则**：每个文件只负责一件事（预处理/检测/识别/重建/输出），模块间通过统一的数据结构（元素字典列表）传递，任何一个模块都可以独立替换或测试。

---

**Q17：系统的输出 data-bbox 坐标是相对坐标还是绝对坐标？能和原始 PDF 对应吗？**

A：输出的 `data-bbox` 是**图像像素坐标**（绝对像素值），基于预处理后的增强图像（`enhanced_*.png`）。

与原始 PDF 的对应关系：
- 若输入是图片，像素坐标直接对应原始图片（增强过程等比缩放，坐标可按缩放比例还原）
- 若输入是 PDF，像素坐标对应 300 DPI 渲染后的图像，可通过 DPI 转换为 PDF 坐标：`pdf_pt = pixel / (300/72)`

当前系统没有实现 PDF 坐标还原的封装，这是一个可扩展的方向——将 `data-bbox` 附加 DPI 信息，下游工具可以精确定位到 PDF 页面中的原始位置，实现文本高亮等功能。

---

**Q18：项目开发过程中遇到的最大坑是什么？**

A：最典型的是 **PPStructure 版面误判**问题（见 Q4 详解）。

开发初期我们完全相信 PPStructure 的输出，认为 `figure` 区域就是图片，不需要从中提取文字。但在实际测试时，信息图风格的文档（含图标装饰的 PPT 截图）完全识别不出文字，输出了空 JSON。

排查花了一些时间才发现，PPStructure 的 `res` 字段里其实有 OCR 识别结果，只是被我们的代码直接忽略了（因为类型是 `figure`）。

这个问题让我们认识到：**不能盲目相信模型的高层分类结果，底层的 OCR 子结果往往更可靠**。修复后的逻辑：先看 `res` 里有没有文字，再决定当成图片还是文本处理。

### 模块四：工程与部署类

**Q19：如何在没有网络的情况下部署这个系统？**

A：系统的网络依赖只有两个：

1. **模型下载**：PaddleOCR/PPStructure 首次运行时会下载模型到 `~/.paddleocr/`。可以在有网络的环境预先下载好，然后把 `~/.paddleocr/` 目录整体拷贝到离线环境。也可以在 PaddleOCR 初始化时通过 `det_model_dir`、`rec_model_dir`、`table_model_dir`、`layout_model_dir` 四个参数指定本地路径。

2. **Mathpix API**：仅公式识别使用，可选。离线时自动降级为本地 OCR，不影响其他功能。

因此，完全离线部署只需要提前下载模型文件，其他代码和依赖均可通过 pip 离线安装。

**Q20：如果要把这个系统做成 Web 服务，你们会怎么设计 API？**

A：用 FastAPI 可以很快实现，大致 API 设计：

```
POST /parse
  Body: multipart/form-data
    file: <PDF 或图片文件>
    format: html | json | both  (可选，默认 both)
  Response: {
    "html": "...",          // HTML 字符串
    "json": {...},          // JSON 对象
    "stats": {
      "total_elements": 12,
      "elapsed_seconds": 2.1,
      "page_count": 1
    }
  }

GET /health
  Response: {"status": "ok", "gpu": false, "ppstructure": true}
```

关键工程考虑：
- 模型在服务启动时预加载到内存，避免每次请求重新初始化（节省 3～5 秒）
- 文件上传大小限制（建议 50MB）
- 多进程/异步处理多并发请求
- 临时文件（tmp/ 目录）定期清理
